# DOCUMENT STANDARD

상위 인덱스: `operations/OPERATIONS_BOOK.md`

목적: 문서 위치/이름/형식을 최소한으로 정규화해서, 사람이 문서를 찾고 유지하기 쉽게 만든다.

## 1. 문서 체계 범위
문서 정규화 대상은 `docs/` 안 문서만이 아니다. 아래의 docs 밖 문서도 포함한다.
- 루트 규칙 문서: `AGENTS.md`, `SOUL.md`, `USER.md`, `MEMORY.md`, `HEARTBEAT.md`
- 운영 문서: `operations/*.md`
- 일반 설명 문서: `docs/*.md`
- 상태 문서: `runtime/*.md` 중 사람이 읽는 문서
- 기록 문서: `memory/*.md`

## 2. 폴더 역할 표준
- `operations/` : 운영 기준서, 운영북, 운영 절차, 운영 인벤토리
- `docs/` : 일반 기능 설명, 참고 문서, 사용자/개발 문서
- `runtime/` : 현재 상태, 재로딩용 상태파일, 실행 중 산출 상태
- `memory/` : 일일 기록, 결정 로그, proof 로그

## 3. 네이밍 표준

### 2-1. 중앙 기준서/문서북
- 형식: `*_BOOK.md`
- 예: `OPERATIONS_BOOK.md`
- 용도: 어떤 영역의 전체 지도, 목차, 읽는 순서, 대상 목록을 제공하는 문서

### 2-2. 폴더 안내 문서
- 형식: `README.md`
- 용도: 해당 폴더의 목적, 먼저 볼 파일, 포함/비포함 범위를 설명

### 2-3. 상태파일
- 형식: 소문자 + 하이픈 + `.md`
- 예: `current-task.md`
- 용도: 프로그램/에이전트가 다시 읽어야 하는 현재 상태

### 2-4. 일일 로그
- 형식: `YYYY-MM-DD.md`
- 용도: 그날의 결정, 이유, 다음 액션, proof 기록

### 2-5. 개별 짧은 기준 문서
- 원칙: 가능하면 별도 파일을 만들지 말고 해당 `*_BOOK.md` 섹션에 흡수한다.
- 예외: 독립 규칙으로 자주 참조되고, 북 안에 넣으면 오히려 찾기 어려울 때만 별도 문서 허용

## 4. 문서북 작성 요령
문서북은 아래 순서를 기본으로 한다.
1) 목적
2) 목차
3) 반드시 읽어야 할 문서
4) 필요 시 읽는 문서
5) 대상 목록
   - 프로그램 목록 / 상태파일 목록 / 절차 목록
6) 상황별 어디를 볼지
7) 갱신 규칙

## 5. 운영 프로그램 표준 항목
문서북 안에서 운영 프로그램을 정리할 때는 아래 칼럼을 우선 사용한다.
- 이름
- 종류
- 역할
- 기본 사용법
- 위치
- 상태
- 관련 문서(선택)

## 6. 상태파일 작성 표준
`runtime/current-task.md` 같은 상태파일은 아래 필드를 기본으로 한다.
- `current_goal`
- `last_completed_step`
- `next_action`
- `required_paths_or_params`
- `notes`

원칙:
- 자유서술보다 필드 중심
- 짧고 재로딩 가능하게 유지
- 장문 히스토리는 넣지 않음

## 7. 로그 문서 작성 표준
`memory/YYYY-MM-DD.md`는 아래 필드를 기본으로 한다.
- `what`
- `why`
- `next`
- `proof`

원칙:
- 기록/검색용
- 재로딩용으로 통째로 읽지 않음
- 중요한 결정/변경만 남김

## 8. 문서 생성 원칙
- 새 문서를 만들기 전에 기존 북/README/상태파일에 흡수 가능한지 먼저 본다.
- 운영 관련이면 `docs/`보다 `operations/`를 먼저 검토한다.
- 이름만 봐도 역할이 드러나야 한다.
- 중복 문서가 생기면 북으로 통합하고 포인터만 남긴다.

## 9. 한 줄 규칙
- `AGENTS.md`는 문서 체계에 포함되지만, 자동 로드되는 상위 규칙으로 분류한다.
- 책은 `*_BOOK.md`
- 안내는 `README.md`
- 상태는 `runtime/*.md`
- 기록은 `memory/YYYY-MM-DD.md`
- 운영은 `operations/`
- 일반 설명은 `docs/`
