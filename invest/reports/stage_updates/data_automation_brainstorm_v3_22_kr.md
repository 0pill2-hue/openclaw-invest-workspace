## 1) kr/supply 1종목 누락 자동복구 루틴

### 옵션 비교

| 옵션 | 개요 | 정확도 | 비용 | 운영복잡도 |
|---|---|---:|---:|---:|
| A. 전체 재수집 | 누락 발생 시 전체 종목 다시 수집 | 높음 | 매우 높음 (시간/API) | 낮음 |
| B. 누락 종목 타깃 복구(선정) | master 대비 누락/빈파일만 탐지 후 해당 종목만 재수집 | 높음 | 낮음 | 중간 |
| C. 이전 파일 복제 복구 | 누락 시 과거 snapshot 복사 | 중간(정합성 낮음) | 낮음 | 중간 |

### 채택안
- **B. 누락 종목 타깃 복구**
- 채택 사유:
  1. 실제 이슈(1종목 누락)와 정확히 맞는 최소비용 대응
  2. 무결성(실수집) 유지, 복제/스냅샷 의존 제거
  3. 크론 주기 내 수행 가능한 실행시간

### Reject 사유
- A reject: 신뢰도는 높지만 운영비용 과다(불필요한 전체 재수집)
- C reject: 빠르지만 데이터 최신성/정확성 훼손 위험

## 2) 백필 완료 시 자동 종료/전환 룰 (DART)

### 옵션 비교

| 옵션 | 개요 | 정확도 | 비용 | 운영복잡도 |
|---|---|---:|---:|---:|
| A. 크론 엔트리 직접 삭제/주석 | 완료 시 crontab 직접 편집 | 높음 | 낮음 | 높음(권한/경합 위험) |
| B. 상태기반 Autopilot + 내부 Throttle(선정) | 크론은 유지, 완료시 monitor 모드 전환 및 실행 간격 완화 | 높음 | 매우 낮음 | 낮음~중간 |
| C. 고정 스케줄 완화(수동) | 사람이 완료 후 수동으로 주기 변경 | 중간 | 낮음 | 낮음(자동화 아님) |

### 채택안
- **B. 상태기반 Autopilot + 내부 Throttle**
- 채택 사유:
  1. crontab 직접수정 없이 자동 전환 가능
  2. 완료 후 즉시 백필 부담 제거(내부 24h monitor tick)
  3. 재결손 발생 시 자동 재활성(backfill 모드 복귀)

### Reject 사유
- A reject: crontab 동시 수정/권한/실패 복구 리스크 큼
- C reject: 자동화 요구 충족 불가(인간 개입 필요)

## 3) RSS 날짜 파싱 예외 자동 보정 + Telegram undated 보정

### 옵션 비교

| 옵션 | 개요 | 정확도 | 비용 | 운영복잡도 |
|---|---|---:|---:|---:|
| A. 수집시 정규화만 적용 | 신규 데이터만 파싱 보정, 과거는 미보정 | 중간 | 낮음 | 낮음 |
| B. 수집시 정규화 + 과거 백필 보정 + undated 자동패치(선정) | 신규/과거 모두 보정 + 텔레그램 undated 자동 헤더 주입 | 높음 | 중간 | 중간 |
| C. 파싱 실패시 전부 파일mtime 대체 | 예외는 사라지나 날짜 정확성 저하 | 낮음~중간 | 낮음 | 낮음 |

### 채택안
- **B. 수집시+과거 일괄 보정 + Telegram undated 자동패치**
- 채택 사유:
  1. 신규/기존 데이터 모두 커버(재발 + 기존부채 동시 해소)
  2. undated 파일을 자동으로 Date 파싱 가능 상태로 전환
  3. fallback source 기록으로 사후 감사 가능

### Reject 사유
- A reject: 기존 적체(undated) 미해결
- C reject: 예외는 줄지만 날짜 정확성 희생

## 최종 선정 요약

1. **Supply 누락 복구**: 타깃 종목 자동복구
2. **DART 백필 전환**: Autopilot + 내부 주기완화(완료 후 monitor)
3. **RSS/Telegram 날짜 보정**: 수집단 정규화 + 배치 보정 + undated 자동패치
