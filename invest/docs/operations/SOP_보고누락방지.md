## 1. 3점 보고 템플릿

| 시점 | 트리거 | 필수 포함 내용 |
|------|--------|----------------|
| **🟢 시작** | 작업 착수 즉시 | `작업명 / 예상 소요 / 리스크` |
| **🟡 중간** | (장기/중요 작업만) 50% 도달 or 30분 경과 | `진행률 / 이슈 / 예상 완료` |
| **🔴 종료** | 결과 확정 직후 | `결과물 / 등급(DRAFT·VALIDATED·PROD) / 후속 필요` |

```
[시작] {작업명} 시작. 예상 {N}분. 리스크: {있음/없음}
[중간] {작업명} {N}% 완료. 이슈: {내용 or 없음}. ETA: {HH:MM}  # 장기/중요 작업만
[종료] {작업명} 완료. 결과: {요약}. 등급: {DRAFT/VALIDATED/PROD}
```

## 2. PENDING_REPORT 큐 운영 규칙

**등록 조건** — 아래 중 하나라도 해당하면 즉시 큐에 추가:
- "보고드리겠습니다" 발화 시
- 작업 착수 시 (시작 보고 전송 전)
- 중간 지연 발생 시

**큐 항목 형식** (`TASKS.md` 상단 섹션에 기록):
```
PENDING_REPORT | {작업명} | 등록: HH:MM | 마감: HH:MM | 상태: OPEN
```

**상태 전이**: `OPEN → SENT → CLOSED`
- OPEN: 보고 미전송
- SENT: 전송 완료, 확인 대기
- CLOSED: 확인 완료 or 불필요 판정

**규칙**:
1. 세션 시작 시 OPEN 항목 전체 확인 필수
2. OPEN 항목 존재 시 → 다음 작업 착수 **허용(병행 가능)**, 단 OPEN 항목 보고 cadence(시작/중간/종료) 유지 필수
3. 큐 비워지지 않으면 heartbeat에서 자동 검출

## 3. Heartbeat 연동 점검 규칙

**매 heartbeat 실행 시 자동 점검 순서:**

```
① TASKS.md → PENDING_REPORT 섹션 스캔
② OPEN 항목 있으면 → 즉시 보고 전송
③ 마지막 보고 시각 확인 → 30분 초과 시 중간 보고 강제 실행
④ 작업 중 세션 중단 흔적 있으면 → 종료 보고 재전송
⑤ 이상 없으면 → HEARTBEAT_OK
```

**`HEARTBEAT.md` 점검 항목 추가 필수:**
```markdown
## 보고 점검
- [ ] PENDING_REPORT OPEN 항목 없음
- [ ] 최근 보고 30분 이내
```

## 4. 실패 시 에스컬레이션 규칙

| 실패 유형 | 1차 조치 | 2차 조치 (5분 내 미해결) |
|-----------|----------|--------------------------|
| 보고 전송 실패 | 재시도 1회 + 큐 유지 | 채널 변경 후 재전송 |
| 30분 이상 중간 보고 누락 | 즉시 중간 보고 전송 | 지연 사유 + 사과 메시지 추가 |
| 종료 보고 없이 세션 종료 | 다음 세션 시작 시 소급 전송 | TASKS.md에 미처리 로그 기록 |
| 큐 항목 72시간 이상 OPEN | 작업 폐기 or 재개 여부 확인 요청 | 주인님께 명시적 의사결정 요청 |

**에스컬레이션 메시지 형식:**
```
⚠️ 보고 지연 알림
작업: {작업명}
원인: {이유}
현재 상태: {상태}
요청: {확인 or 결정 사항}
```

---

> **핵심 원칙**: 보고 없는 완료는 완료가 아니다. 병행은 가능하되, OPEN 보고는 반드시 cadence를 지키며 닫는다.
