# RULEBOOK V1 (2026-02-18)

## 1) 운영 구조 (확정)
- 운영 방식: **3검증 1운영**
- 실매매 운영 전략: **1개만 집행(현재 Hybrid)**
- 나머지 2개(Quant/Text): 검증·감시 전용(매매 미집행)

## 2) 전략 구조 룰 (확정)
1. 메인 신호: 이웃지표
2. 내지표: 점수화 금지, 필터/가드로만 사용
3. 동일 지표는 동일 정의/계산식/스케일 강제

## 3) 트랙 역할 (확정)
- Quant: 가격/수급 이상탐지 레퍼런스
- Text: 이벤트 조기신호 레퍼런스
- Hybrid: 최종 집행 엔진

## 4) 충돌 해소 룰 (신규 확정)
- 최종 의사결정 우선순위: Hybrid
- Quant/Text는 거부권이 아닌 경고 플래그
- 경고 누적 시: 포지션 축소/보류 트리거

## 5) 저회전 룰 (확정)
- 최소 보유 20거래일
- 리밸런싱 주기: 주 1회
- 교체 임계치: 기존 대비 +15% 우위일 때만
- 월간 교체 상한: 보유 종목의 30%
- 분할 교체: 50% → 100% 2단계
- 보유 유지 우선(탈락점 기준 관리)

## 6) 매매/리스크 룰 (확정)
- 왕복 비용 패널티: 3~4% 보수 가정
- 일평균 거래대금 100억 이하 단계적 패널티
- 보유 종목수 1~6개 제한
- 리스크 급증 시 현금 100% 전환 가능
- 거버넌스 리스크 종목 컷

## 7) 거버넌스 룰 (확정)
- 결과 등급: DRAFT | VALIDATED | PRODUCTION
- DRAFT는 학습/평가 입력 금지
- 경로 분리: test / validated / prod
- Purged CV/OOS 통과 전 채택 금지
- 중요 산출물 교차리뷰 통과 전 제출 금지

## 8) 단계 운영 룰 (확정)
- 11단계 체계로 진행
- 단계 보고는 처음부터 끝까지 나열
- 단계별 진행률 표기
- 단계 완료 시 Git 업데이트 + 완료보고 필수

## 9) 승격/강등 룰 (신규 확정)
- Quant/Text는 기본 감시 레이어
- OOS 조건 충족 시 운영 후보 승격 검토
- 승격 판단은 7단계 게이트 이후만 허용

## 10) 탈락 기준 v1 (보수형, 확정)
### 10-1. Hard Drop (즉시 탈락)
- 거버넌스 위반:
  - DRAFT 모델의 운영 투입
  - 7단계(Purged CV/OOS) 미통과 모델 채택 시도
  - 등급/보고 위반(PRODUCTION 미만을 공식 성과로 보고)
- 성능/리스크 위반:
  - Hybrid MDD > 25%
  - Quant/Text MDD > 30%
  - Rolling 3M Sharpe < -0.1 (Hybrid), < -0.2 (Quant/Text)
  - Rolling 3M 초과수익 < -15% (Hybrid), < -18% (Quant/Text)
- 운영 무결성 위반:
  - 데이터 결측/입력 차단(DC-01)이 연속 3회 발생 후 미복구
  - 분기 교체 1회 상한 위반

### 10-2. Soft Warning (누적 경고)
- 1회 경고 = 1점, 8주 누적 3점 시 탈락 검토 의무
- Soft 트리거(예):
  - Rolling 3M Sharpe < 0.2 (2개월 연속)
  - MDD 18~25% 구간 4주 지속(Hybrid)
  - 주간 승률 < 42% 3주 연속
  - 변동성 기준 대비 1.5배 4주 지속
- 복구 규칙: 경고 후 8주 무재발 시 1점 소멸

### 10-3. 교체 의사결정 안전밸브
- Soft 3점 누적 후에도 즉시 교체하지 않는다.
- `기대 개선효과 > 스위치 총비용(약 18~19%)`일 때만 교체 집행.
- 미충족 시 유지 + 포지션 축소/보류로 전환.

## 11) 적용 우선순위
1. 본 RULEBOOK
2. stage 문서(stage04 등)
3. 개별 스크립트 구현

위 우선순위 충돌 시 상위 문서를 따른다.

## 12) 리팩토링 원칙 (동작동일성, 단계적 이전)
- 리팩토링의 1차 목표는 구조 개선이며, 기능/CLI/출력 포맷 변경은 금지한다.
- 코드 이동은 순수 함수 단위로 최소 침습 수행하고, 기존 엔트리포인트는 유지한다.
- 이전은 단계적으로 진행하며 각 단계마다 컴파일/스모크 검증을 통과해야 한다.
- 검증 실패 시 즉시 원복 후 원인과 재시도 계획을 먼저 문서화한다.
