# CONTRIBUTING

GitHub/작업 브랜치/커밋/PR 규칙의 canonical 문서.

## 브랜치/머지 정책
- `main` 직접 푸시 금지
- 기능 브랜치에서 작업
- `main` 반영은 PR만 허용

## 커밋 룰
- 형식: `type: summary`
- 커밋 전 체크:
  1. 민감정보 포함 여부
  2. 실행/문법 검증 통과 여부
  3. TASKS/메모리 업데이트 필요 항목 반영

## Push 타이밍 정책
1. 커밋 직후 지체 없이 원격 push
2. push 대상은 현재 작업 브랜치, `main` 직접 push 금지
3. 동일 작업에서 커밋 2개 이상 누적 시 마지막 커밋 직후 즉시 push

예외:
- 민감정보 누출 점검 미완료
- 대용량 산출물 정리/ignore 정합성 미완료

## 코드 적용 프로세스
1. 이슈/작업 정의
2. 기능 브랜치 생성
3. 구현 + 로컬 검증
4. PR 생성
5. 중요 변경 교차리뷰 1회
6. CI 통과
7. 승인 후 merge
8. 배포/운영 반영
